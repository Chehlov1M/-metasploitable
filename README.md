## Задание 2. Режимы сканирования Nmap и анализ трафика

### Чем отличаются эти режимы сканирования с точки зрения сетевого трафика?

Режимы сканирования Nmap принципиально различаются по типу отправляемых пакетов, логике взаимодействия и видимости в сети:

*   **SYN-сканирование (`-sS`)** — это «полуоткрытое» сканирование. Nmap отправляет пакет с флагом `SYN` для инициации соединения. Ключевая особенность трафика: если порт открыт, сервер отвечает `SYN-ACK`, но Nmap **не завершает** рукопожатие, а сразу шлёт пакет с флагом `RST` (сброс соединения). Это делает сканирование быстрым и относительно скрытным, так как сервисы на целевой машине не успевают зафиксировать полноценное подключение. В Wireshark это выглядит как чёткие пары пакетов `SYN → SYN-ACK → RST`.
*   **FIN-, Xmas-сканирования (`-sF`, `-sX`)** используют нестандартные TCP-флаги. FIN-сканирование отправляет пакет только с флагом завершения `FIN`, а Xmas («рождественское») — сразу с тремя флагами: `FIN + PSH + URG`. Согласно RFC 793, закрытые порты должны ответить `RST`, а открытые — игнорировать такие пакеты. В трафике эти режимы легко детектируются по аномальным комбинациям флагов. Xmas-пакеты особенно заметны и часто блокируются системами обнаружения вторжений (IDS).
*   **UDP-сканирование (`-sU`)** работает принципиально иначе, поскольку UDP не устанавливает соединений. Nmap отправляет пустой или специфичный UDP-пакет на целевой порт. Трафик здесь характеризуется не TCP-флагами, а наличием ICMP-сообщений. Кроме того, из‑за необходимости ждать таймаута (чтобы понять, что ответа не будет), UDP-сканирование генерирует длинные паузы между пакетами, что сильно отличает его по временной структуре от быстрых TCP-режимов.

### Как отвечает сервер (Metasploitable 2)?

Реакция сервера напрямую зависит от состояния порта (открыт/закрыт) и используемого протокола сканирования:

| Режим сканирования | Если порт **открыт** | Если порт **закрыт** |
| :--- | :--- | :--- |
| **SYN (`-sS`)** | Сервер отвечает пакетом `SYN-ACK` (подтверждает готовность к соединению). Nmap сразу разрывает сессию пакетом `RST`. | Сервер сразу отвечает пакетом `RST` (отказ в соединении). |
| **FIN (`-sF`)** | Сервер **не отвечает** (игнорирует пакет). Это поведение соответствует спецификации стека TCP в Unix/Linux. | Сервер отвечает пакетом `RST` (сбрасывает некорректный запрос). |
| **Xmas (`-sX`)** | Сервер **не отвечает**. Аналогично FIN-сканированию, Linux-системы молча отбрасывают пакеты с нестандартными флагами на открытые порты. | Сервер отвечает пакетом `RST`. |
| **UDP (`-sU`)** | Сервер **не отвечает**, либо присылает специфический ответ от сервиса (например, баннер DNS или SSH). Отсутствие ответа интерпретируется Nmap как «открыт», так как UDP не обязан подтверждать получение. | Сервер присылает ICMP-сообщение типа `Port Unreachable` («порт недоступен»). Именно по этому ответу Nmap понимает, что порт закрыт. |

### Подтверждение результатами анализа в Wireshark

При анализе файлов захвата (`.pcapng`) в Wireshark различия видны наглядно:

1.  **В `syn_scan.pcapng`** чётко прослеживается шаблон незавершённого соединения: для открытых портов видна цепочка из трёх пакетов (`SYN`, `SYN‑ACK`, `RST`), а для закрытых — всего два (`SYN`, `RST`).
2.  **В `fin_scan.pcapng` и `xmas_scan.pcapng`** для открытых портов в трафике **нет ответа от сервера**. Nmap делает вывод об открытом порте именно по этой «тишине». При этом в деталях пакета Xmas явно видны установленные флаги `FIN`, `PSH` и `URG` — в легитимном трафике такая комбинация практически не встречается.
3.  **В `udp_scan.pcapng`** преобладают ICMP-сообщения `Type 3 Code 3` (Port Unreachable) в ответ на запросы к закрытым портам. Между отправками UDP-пакетов заметны значительные временные интервалы (таймауты), что объясняет низкую скорость этого режима.

Таким образом, каждый режим сканирования оставляет уникальную «сигнатуру» в сетевом трафике, которую можно однозначно идентифицировать и проанализировать с помощью Wireshark, подтвердив теоретические принципы работы протоколов TCP и UDP.
